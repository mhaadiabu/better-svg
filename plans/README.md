## Findings considered and rejected (second run, 2026-06-25)

The second audit pass (focused on `@mhaadi/svg` Socket quality drop and
the React adapter reparse issue) identified six findings. Five became
plans (009, 010, 011) or were absorbed into existing plans. The
following were considered and rejected:

- **Direction D2 — `fetchOptions` latest-ref pattern.** Treat
  `fetchOptions` like the callback ref pattern from plan 006. **Rejected**
  for the perf plan (009): it changes user-visible network semantics
  (a new `fetchOptions` should arguably trigger a refetch to apply new
  headers/cache directives). The 009 plan keeps `fetchOptions` in the
  effect deps by design; the parsed-cache fix already makes the
  re-trigger cheap (no DOMParser re-run). If a future consumer reports
  the re-trigger as a real problem, re-open as a follow-up plan.
- **Direction D4 (precursor) — TTL/time-based eviction.** Replace the
  LRU with a TTL or TTL+LRU hybrid. **Rejected for 011**: time-based
  eviction requires a per-entry timestamp and a sweep loop, which is
  a bigger change than the LRU cap. The maintainer can add TTL later
  if the size cap is observed to hold hot entries past their useful
  life. The LRU alone resolves the unbounded-growth concern.
- **Unify the four adapter-local `Map<string, string>` caches into
  one shared module instance as a pre-009 prerequisite.** Rejected
  because 009's parsed-cache move to `core/cache.ts` *is* the
  unification. Doing it as a prerequisite would have been two
  refactors for the same outcome. 009 is the right place.
- **Drop the `devDependencies` from the published `package.json`
  entirely (instead of relying on `files: ["dist"]` to exclude them
  from the tarball).** Rejected for 010: `devDependencies` are the
  standard way to declare test/build tooling without forcing
  consumers to install them. npm honors `files` for the tarball but
  exposes the metadata on the registry. The right fix is what 010
  does: clean up other metadata signals (LICENSE, repository,
  engines, etc.) so Socket weights the visible metadata fairly.
  Removing `devDependencies` would break `pnpm install` from the
  repo and the CI pipeline that runs `pnpm install --frozen-lockfile`.
- **Switch the React adapter from `dangerouslySetInnerHTML` to a
  real React tree of `svg`/`path`/`circle`/etc. components (à la
  `native.tsx:212-228`).** Rejected: the React adapter's whole point
  is "render any SVG inline without a per-element component map."
  The current `dangerouslySetInnerHTML` approach trades XSS surface
  (mitigated by `parseInlineSvg` sanitization) for a tiny bundle and
  compatibility with arbitrary SVGs. Changing this is a public-API
  decision, not a perf fix.
- **Replace `DOMParser` with a hand-rolled tokenizer for the parse
  path to remove the DOM cost entirely.** Rejected: `DOMParser` is
  browser-native, well-tested, and the cost is a one-time parse per
  source (cached after 009). Hand-rolling a tokenizer would be a
  significant new code path with its own bug surface, and the
  `parseWithRegex` fallback in `ast.ts:83-148` already exists for
  non-DOM environments. The right knob is to make the parse run
  less often (009), not to make the parse faster.
